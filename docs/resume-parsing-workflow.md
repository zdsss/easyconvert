# 简历解析 Workflow 设计

## 1. 解析流程设计

### 1.1 核心流程

```
文件接收 → 格式识别 → 内容提取 → 结构化解析 → 验证 → 输出
```

### 1.2 各阶段详细设计

#### 阶段1: 文件接收
**输入**: PDF/Word/图片文件
**处理**:
- 文件格式验证（MIME type检查）
- 文件大小检查（<10MB建议，>10MB警告）
- 文件完整性验证
- 生成唯一任务ID

**输出**: 文件元数据 + 任务ID

#### 阶段2: 格式识别
**输入**: 原始文件
**处理**:
- PDF版本检测
- 是否包含图片/表格/多栏布局
- 文本可提取性评估
- 编码检测

**输出**: 文件特征标签

#### 阶段3: 内容提取
**输入**: 原始文件 + 特征标签
**处理**:
- PDF文本提取（pdf-parse）
- 图片OCR（如需要）
- 表格结构识别
- 布局分析

**降级策略**:
- 文本提取失败 → OCR
- OCR失败 → 返回错误，标记为"需人工处理"

**输出**: 原始文本 + 结构信息

#### 阶段4: 结构化解析
**输入**: 原始文本 + 结构信息
**处理**:
- 模块识别（基本信息、教育、工作、技能等）
- 字段提取
- 时间格式标准化
- 关键词提取

**输出**: 结构化JSON数据

#### 阶段5: 验证
**输入**: 结构化数据
**处理**:
- 必填字段检查
- 数据格式验证
- 逻辑一致性检查
- 质量评分

**输出**: 验证报告 + 质量分数

#### 阶段6: 输出
**输入**: 验证后的数据
**处理**:
- 格式转换（JSON/Markdown）
- 数据清洗
- 敏感信息脱敏（如需要）

**输出**: 最终结果

## 2. 基于分类的差异化处理

### 2.1 按模块完整度分类

#### Simple（≤3个模块）
- **特点**: 基础信息为主
- **策略**: 快速提取，重点关注基本信息和联系方式
- **超时**: 5秒
- **验证**: 宽松模式，只验证必填字段

#### Standard（4-5个模块）
- **特点**: 标准简历结构
- **策略**: 标准流程，全模块解析
- **超时**: 10秒
- **验证**: 标准模式，验证常见字段

#### Complete（≥6个模块）
- **特点**: 内容丰富，模块多样
- **策略**: 深度解析，识别所有模块
- **超时**: 15秒
- **验证**: 严格模式，全面验证

### 2.2 按内容详细度分类

#### Brief（<50行）
- **处理**: 单次解析，无需分段
- **内存**: 低（<2MB）
- **并发**: 高（10个/批次）

#### Normal（50-150行）
- **处理**: 标准解析
- **内存**: 中（2-5MB）
- **并发**: 中（5个/批次）

#### Detailed（>150行）
- **处理**: 分段解析，避免token超限
- **内存**: 高（5-10MB）
- **并发**: 低（2个/批次）

### 2.3 特殊情况处理

#### 图片简历
- 触发条件: 无法提取文本或文本<100字符
- 处理策略:
  1. 使用OCR提取文本
  2. 图片预处理（去噪、二值化）
  3. 多语言识别
- 性能影响: +3-5秒

#### 表格简历
- 触发条件: 检测到表格结构
- 处理策略:
  1. 表格边界识别
  2. 单元格内容提取
  3. 行列关系映射
- 性能影响: +1-2秒

#### 多栏布局
- 触发条件: 检测到多栏布局
- 处理策略:
  1. 栏目分割
  2. 阅读顺序重建
  3. 内容合并
- 性能影响: +1-2秒

## 3. 验证节点设计

### 3.1 解析完整性验证

**验证点**: 阶段4完成后
**检查项**:
- [ ] 至少识别出1个模块
- [ ] 提取的文本长度 > 50字符
- [ ] 模块数量与预期相符（±1）

**失败处理**: 标记为"解析不完整"，记录缺失模块

### 3.2 数据准确性验证

**验证点**: 阶段5
**检查项**:
- [ ] 姓名格式正确（2-4个汉字或英文名）
- [ ] 手机号格式正确（11位数字）
- [ ] 邮箱格式正确
- [ ] 日期格式标准化
- [ ] 教育/工作经历时间线合理

**失败处理**: 标记字段为"待确认"，不阻断流程

### 3.3 格式规范性验证

**验证点**: 阶段6输出前
**检查项**:
- [ ] JSON结构符合schema
- [ ] 所有字段类型正确
- [ ] 枚举值在允许范围内
- [ ] 无重复数据

**失败处理**: 自动修正或返回错误

### 3.4 性能指标验证

**验证点**: 全流程结束后
**检查项**:
- [ ] 总耗时 < 预期超时时间
- [ ] 内存使用 < 限制
- [ ] API调用次数 < 限制

**失败处理**: 记录性能日志，触发告警

## 4. 性能优化方案

### 4.1 缓存策略
- **文件级缓存**: 相同文件MD5，直接返回缓存结果
- **模块级缓存**: 相同文本块，复用解析结果
- **过期时间**: 24小时

### 4.2 并发控制
- **按分类分组**: 不同类型简历使用不同并发数
- **动态调整**: 根据系统负载自动调整
- **队列管理**: 优先处理小文件

### 4.3 资源限制
- **内存限制**: 单个任务最大100MB
- **超时控制**: 按分类设置不同超时
- **降级机制**: 超时后返回部分结果

### 4.4 批处理优化
- **批量提取**: 一次性提取多个文件的文本
- **批量解析**: 合并API调用
- **流式处理**: 边提取边解析

## 5. 错误处理和降级策略

### 5.1 错误分类

| 错误类型 | 处理策略 | 用户提示 |
|---------|---------|---------|
| 文件损坏 | 返回错误 | "文件无法读取，请检查文件完整性" |
| 格式不支持 | 返回错误 | "不支持的文件格式" |
| 提取失败 | 降级到OCR | "正在使用OCR识别..." |
| OCR失败 | 返回错误 | "无法识别文件内容" |
| 解析超时 | 返回部分结果 | "解析超时，返回部分结果" |
| API限流 | 重试3次 | "服务繁忙，请稍后重试" |

### 5.2 降级路径

```
标准解析 → 简化解析 → 基础提取 → OCR → 失败
```

### 5.3 重试机制
- **网络错误**: 指数退避重试，最多3次
- **临时错误**: 立即重试1次
- **永久错误**: 不重试，直接返回

## 6. 监控和日志

### 6.1 关键指标
- 成功率（目标: >95%）
- 平均耗时（目标: <10秒）
- P95耗时（目标: <20秒）
- 错误率（目标: <5%）

### 6.2 日志记录
- 每个阶段的耗时
- 错误详情和堆栈
- 文件特征和分类
- 验证结果

### 6.3 告警规则
- 成功率 < 90%
- 平均耗时 > 15秒
- 错误率 > 10%
- 队列积压 > 100

## 7. 测试策略

### 7.1 单元测试
- 每个阶段独立测试
- 模拟各种输入场景
- 覆盖率 > 80%

### 7.2 集成测试
- 端到端流程测试
- 使用真实简历样本
- 覆盖所有分类组合

### 7.3 性能测试
- 压力测试（100份并发）
- 长时间运行测试（1000份）
- 内存泄漏检测

### 7.4 回归测试
- 固定测试集（30份代表性样本）
- 每次发布前执行
- 对比历史结果
