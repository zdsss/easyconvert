## 二、全局导航和布局框架

### 2.1 布局方案：左侧边栏导航

设计决策：选择左侧可折叠边栏，而非顶部导航。

理由：
1. 三个模块（解析评测API）各有 2-5 个子页面，顶部导航放不下且会变成二级下拉，增加点击次数
2. 左侧边栏在 Vercel、Stripe、Linear 等产品中已是 B2B 工具的事实标准，用户心智模型成熟
3. 可折叠设计在不需要导航时让出更多内容空间（评测详情等数据密集页面）
4. 当前只有两个入口缺少层级感的问题自然解决

### 2.6 交互说明

 交互  行为 
------------
 侧边栏折叠展开  点击底部按钮或拖拽边缘，平滑过渡 `transition-all duration-200`，宽度 240px ↔ 64px 
 模块组折叠  点击模块标题行，子菜单 slide-down `max-h-0 → max-h-200 transition-all duration-200` 
 当前页指示  左侧 2px primary 竖条 + 浅蓝背景，字体 medium 
 通知铃  下拉面板显示最近 5 条，全部已读按钮，点击跳转对应页面 
 全局搜索  Modal 弹出，搜索文件名任务 ID，回车跳转 
 暗色模式  `next-themes`，切换时 200ms 淡入淡出过渡 

### 2.7 设计理由

1. 左侧边栏而非顶部 Tab：解决了导航只有两个入口缺少层级感的问题，且可容纳未来更多模块
2. 可折叠：评测详情等数据密集页面需要最大化内容空间
3. 全局搜索 Cmd+K：参考 Linear 的快捷键优先理念，快速定位文件和任务
4. 面包屑而非二级导航：用户始终知道自己在哪，且不占用额外空间

## 三、解析主页（重新设计）

### 3.1 核心设计决策

问题：当前把单文件上传、批量处理、监控仪表盘塞在一个 Tab 页里。

解决方案：拆分为侧边栏三个独立页面：
- `` 解析简历（单文件上传 + 结果展示）
- `batch` 批量处理
- `monitor` 监控仪表盘

理由：每个功能目标用户不同（HR 用单文件，运营用批量，DevOps 用监控），混在一起增加认知负担。侧边栏有足够的层级空间承载。

## 八、评测详情页（重点重新设计）

### 8.1 核心设计决策

问题：当前信息过载——上传区 + 进度条 + 指标卡 + 筛选器 + 结果表格 + 展开详情 + 弹窗，全部堆在一个长页面。

解决方案：采用渐进式披露 (Progressive Disclosure) + 分层视图

架构：
1. 一级视图（概览）：任务信息 + 整体指标 + 文件上传入口
2. 二级视图（结果列表）：筛选 + 表格 + 分页
3. 三级视图（单条详情）：侧边抽屉，展示对比 + 标注编辑

理由：
- 用户 80% 的时间在查看结果列表，20% 在查看单条详情——侧边抽屉避免页面跳转
- 展开行方式（当前方案）在数据多时导致表格行高不一致，阅读困难
- 新页面方式太重，来回切换效率低

### 8.2 一级视图 — 概览

### 8.7 结果表格优化

设计决策：服务端分页，每页 20 条。

理由：
- 批量评测可能有数百条记录，全部渲染会导致性能问题
- 虚拟滚动增加复杂度，且用户通常需要翻页定位
- 分页配合排序和筛选，信息获取效率更高

表格规范：
```

│  1  │ 张三_前端.pdf   │ [中等] │ 96.2% ▲  │ 92.1%  │ [→] │
│  2  │ 李四_后端.pdf   │ [困难] │ 91.8%     │ 88.5%  │ [→] │
└──────────────────────────────────────────────────────────┘

- 行高 h-12 (48px)
- Hover bg-surface-raised
```

## 十三、暗色模式

### 13.1 实现策略

- 使用 `next-themes` 的 `class` 策略
- CSS 变量定义在 `root` 和 `.dark` 选择器下
- 切换时有 200ms 的过渡动画

### 13.2 关键适配点

```
组件                  Light → Dark 适配
───────────────────────────────────────────────────────────
卡片面板             bg-white → bg-slate-800
边框                 border-slate-200 → border-slate-700
主文本               text-slate-900 → text-slate-50
辅助文本             text-slate-600 → text-slate-400
输入框               bg-white → bg-slate-900
上传区               bg-slate-50 → bg-slate-80050
图表                 浅色网格 → 深色网格
阴影                 减淡透明度，增加模糊度
Canvas (游戏等)      需要单独处理背景色
```

## 十四、设计总结

### 核心设计原则回顾

 原则  实现方式 
---------------
 合理性  信息架构重构：3 模块拆分为独立路由，评测详情采用渐进式披露 
 可用性  每个操作 ≤ 3 步，侧边抽屉避免页面跳转，全局搜索 Cmd+K 
 流畅性  Framer Motion 动画、状态过渡、骨架屏 loading、Toast 反馈 

### 解决的问题清单

 #  原问题  解决方案 
--------------------
 1  解析主页功能混杂  拆分为 3 个独立页面 
 2  评测详情信息过载  渐进式披露 + 侧边抽屉 + 服务端分页 
 3  API 管理无入口  新增 API 管理模块（3 个子页面） 
 4  导航缺少层级感  左侧边栏 + 模块分组 + 面包屑 
 5  展开行不明显  改为侧边抽屉，取消展开行 
 6  无 loading 状态  全套骨架屏组件 
 7  无操作确认  确认对话框组件（高风险要求输入确认） 
 8  无 toast 反馈  Toast 通知系统（4 种类型） 
 9  上传后无预览  文件预览卡片 + 进度管线 
 10  标注编辑器字段少  完整字段编辑器 + 分类折叠 + 添加按钮 
 11  emoji 当图标  Lucide SVG Icons 
 12  无设计系统  完整 Design Tokens 
 13  进度条太细  h-2 rounded-full + 8 阶段管线 
 14  无响应式  完整断点策略 + 移动端适配 
 15  无空状态设计  通用空状态组件 
 16  无暗色模式  next-themes + 完整 Dark Mode tokens
